##### [SEL-01] 온보딩 / 계정 연동

**[기술] 네이버 멀티셀러 IP 제한**

네이버 커머스 API는 애플리케이션당 허용 IP를 최대 3개로 제한합니다. 셀러 100명의 토큰을 중앙 서버 하나에서 관리하는 현재 설계로는 **네이버 공식 솔루션 파트너 등록 없이는 멀티셀러 구조 구현이 불가능합니다.** MVP를 네이버 단일 채널로 좁히더라도 파트너 등록 절차가 선결 과제입니다.

개선안: Week 0에 네이버 커머스 솔루션 파트너 등록 신청 병행. 파트너 등록 전까지는 셀러 수를 3명 이하로 제한하거나, 셀러별 개인 앱 등록 후 토큰을 플랫폼에 위탁하는 방식으로 운영.

**[법무] 셀러 API 키 수탁 처리**

셀러의 네이버 API 키를 중앙 서버에서 대신 호출하는 구조는 개인정보보호법 제26조에 따라 **수탁자 계약과 이용자 동의**가 필요합니다. AWS KMS 등으로 기술적 암호화를 해결해도 위수탁 동의는 별개의 법적 요건입니다.

개선안: 온보딩 화면에 "API 키 수탁 처리 동의" 단계를 명시적으로 추가. 위수탁 계약서 자동 발송 기능 포함.

##### [INT-11] 승인 큐 (Command Approval)

**[보안] 고액 송금 승인의 2FA 필요성**

1,000만원 이상 송금 승인이 단순 클릭 한 번으로 처리될 경우 계정 탈취 시 금융사고가 발생합니다. TradingLab 문서에서도 "금융사고 리스크"를 명시적으로 경고했습니다.

개선안: 1,000만원 이상 건은 **2FA(OTP 또는 SMS 인증) 필수** 적용. 계좌 화이트리스트 외 신규 계좌 송금은 추가 확인 단계 삽입. 승인 이력 전체를 감사 로그(Audit Log)에 불변 기록.

**[UX] SLA 타이머 및 에스컬레이션**

승인 큐에 SLA가 없으면 운영자가 부재 시 거래가 무기한 대기됩니다.

개선안: 승인 유형별 SLA 정의 (예: 비딩 메일 발송 = 2시간, 송금 = 4시간). 타이머 만료 시 Slack 에스컬레이션 알림 자동 발송. 대리 승인자 지정 기능 포함.

---

##### [INT-10] 운영 모니터링 대시보드

**[기술] 실시간 WebSocket 연결**

5분 폴링 방식은 모니터링 대시보드에 부적합합니다. 장애 발생 시 최대 5분 후에야 인지합니다.

개선안: 모니터링 대시보드는 **WebSocket 실시간 연결**로 구현. 주문 건수·API 상태·에이전트 실행 상태는 초 단위 갱신. 장애 감지 시 대시보드 상단에 즉시 배너 표시 및 Slack 알림 병행.

---

##### [INT-05] 시세 조회·비교

**[기술] KAMIS 실시간성 오해 방지**

판매자 화면과 동일하게, 운영자도 KAMIS 데이터가 실시간이 아님을 인지해야 합니다. 이상치 판단의 기준선이 "어제 오전 기준"인지 모르면 잘못된 승인 판단으로 이어집니다.

개선안: 비교표에 데이터 기준 시각을 항상 표시. ±10% 이상치 경고 배너에도 "KAMIS 전일 기준 시세 대비" 문구 명기.

---

##### [INT-12] 클레임 최종 판단

**[UX] AI 초안과 최종 응답 영역 분리**

AI가 생성한 응답 초안을 그대로 발송하면 법적 리스크가 있습니다. 운영자가 AI 초안을 보고 그대로 승인 클릭하는 패턴이 굳어지면, AI 초안의 오류가 그대로 고객에게 전달됩니다.

개선안: AI 초안 영역과 최종 응답 입력 영역을 시각적으로 명확히 분리 (색상·구분선). 초안을 복사 후 수정하는 플로우를 기본값으로 설계하여, 원본 그대로 발송하려면 추가 확인 단계를 거치도록 구현.

**[법무] 식품 위생사고 긴급 처리**

벌레·이물질·부패 등 식품 위생사고는 CS 분류 중 가장 법적 리스크가 높은 케이스입니다.

개선안: AI 1차 분류에서 "식품 위생" 카테고리 감지 시 승인 큐에 **긴급(RED) 배지**로 최상단 노출. 해당 건은 일반 CS 응답 플로우와 분리하여 운영자·품질관리자 동시 알림 발송.

---

##### [INT-09] 운영팀 직접 셀러 운영 이해충돌

**[법무] Phase 1 콜드 스타트 구조의 법적 리스크**

TradingLab 문서의 Phase 1에서 "운영팀이 직접 셀러 계정 5~10개를 운영"하는 방식은 플랫폼 운영팀이 자사 플랫폼에 직접 입점하는 구조로, 공정거래법·온라인 플랫폼 공정화법상 이해충돌 고지 의무가 발생할 수 있습니다.

개선안: 법무 검토 완료 전까지 운영팀 직접 운영 계정은 별도 표시 처리. 화면 설계 시 "운영팀 계정" 구분 필드를 DB·UI 모두에 사전 반영.
