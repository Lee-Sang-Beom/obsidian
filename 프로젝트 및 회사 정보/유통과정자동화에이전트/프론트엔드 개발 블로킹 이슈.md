### [C-02] 개인정보보호법 — 쿠팡 API 개인정보 제한 강화 및 위수탁 계약 필수 ⛔ 공급자 주문 화면 블로킹

> **v1 대비 변경**: 쿠팡 개인정보 대규모 유출 사건(2025.11, 3370만 계정)과 이후 쿠팡 API 개인정보 축소 조치를 반영하여 실질적 위험 수준 대폭 상향.

**현황 (2026년 4월 기준)**
- 쿠팡은 이미 API 응답에서 개인정보를 단계적으로 제거하고 있다. 2024년 8월부터 발주서 조회 API 응답에서 구매자 이메일을 빈값으로 변경했고, 2024년 9월부터는 교환요청 목록에서 실번호를 안심번호로 대체했으며, 2026년 3월에는 고객 문의 API 응답에서 `buyerEmail` 필드 자체를 삭제했다. 2025년 11월 쿠팡 3370만 계정 유출 사건 이후 개인정보보호위원회·과기정통부의 감독이 극도로 강화된 상황이다.
- 공급자에게 주문을 배정할 때, 고객의 배송지·이름·연락처 등의 정보가 거래/정산/이력 과정에서 플랫폼 중앙 PostgreSQL을 경유하는 구조에서는 두 층위의 법적 의무가 발생한다.
	- **① 쿠팡·네이버 → 플랫폼** 구간: 플랫폼이 오픈마켓 API로 고객 정보를 수신하여 저장하는 행위는 개인정보보호법 제26조상 처리위탁에 해당하므로, 쿠팡·네이버와 서면 위수탁 계약이 없거나 API 이용약관 내 허용 조항이 확인되지 않는 한 적법한 처리 근거가 없다. 
	- **② 플랫폼 → 공급자** 구간: 공급자는 플랫폼의 주문 이행 업무를 대행하므로 이 역시 처리위탁에 해당할 가능성이 높으며, 공급자와의 별도 서면 위탁계약도 필요하다. 만약 공급자가 해당 정보를 자신의 독립적 목적에 사용한다고 판단될 경우 제3자 제공으로 전환되어 과징금(매출액 3%)·형사처벌 대상이 된다.

**영향 범위**: SUP-06 주문 수신, SUP-08 출고 처리 — 배송 정보가 표시되는 모든 공급자 화면

**왜 설계를 바꿔야 하는가**
- 단순 안내 문구로 해결되는 문제가 아니다. 쿠팡 유출 사건 이후 감독당국이 배송지 정보 유통 경로를 집중 조사하고 있는 시점에서, 계약 없이 배송지 정보를 중앙 서버에 적재하는 행위는 법적 책임이 매우 크다. 또한 쿠팡이 API에서 개인정보를 지속 제거하는 추세이므로, 중앙 서버 경유 구조로 설계하더라도 향후 API 변경으로 데이터를 받지 못하는 상황이 올 수 있다.

**해결 방향**
- 배송지 정보를 중앙 서버에 저장하지 않고, 공급자 엣지 박스가 오픈마켓 API에서 직접 조회하는 방식으로 아키텍처 재설계.
- 쿠팡·네이버 API 이용약관 내 배송 목적 데이터 처리 허용 조항 유무를 우선 확인하고, 허용 조항이 없을 경우 각 플랫폼과 서면 위수탁 계약 체결. 공급자와도 주문 이행 목적의 개인정보 처리위탁 계약을 별도로 체결해야 한다.
- 계약 체결 전까지 SUP-06·08 화면에서 배송지 상세 정보(이름, 주소, 연락처) 노출 불가. 주문번호·상품명만 표시하는 제한 버전으로 개발.

### [C-03] 공정거래법 — 비딩 구조의 정보교환 담합 위험 ⚠️ 공급자 대면 화면 설계 주의

2021년 전면 개정 공정거래법은 경쟁사업자 간 가격 정보교환을 담합의 독립 유형으로 신설했으며, 플랫폼 등 중간매개자를 거쳐 이루어지는 간접 정보교환도 규율 대상에 포함된다. OpenAnt가 공급자들의 입찰가를 수집하고 이를 다시 개별 공급자 화면에 노출하는 구조는, 플랫폼이 허브 역할을 하는 Hub-and-Spoke 정보교환 경로가 될 수 있다.

단, 담합의 직접 당사자는 공급자들이며 OpenAnt는 중간매개자 지위다. OpenAnt에 대한 직접 제재보다는 플랫폼 설계 변경 요구 및 조사 협조 의무가 현실적 리스크다.

**화면별 위험 수준**
- **SUP-11 공급자 성과 대시보드**: 타사 응찰가·가격 순위가 노출되면 개별 사업자를 특정할 수 있는 비공개 가격 정보의 교환 경로가 된다. **⛔ 위험**

**영향 범위**: SUP-11 성과 조회 내 타사 가격·순위 표시 영역, INT-04·05의 공급자 접근 권한 설계

**해결 방향**
- SUP-11에서 타사 응찰가·가격 순위 완전 차단. 자신의 응찰 이력과 낙찰 여부만 표시.
- INT-04·05는 운영자 전용 접근으로 권한 통제 확인. 공급자 계정으로 해당 화면 접근 불가 구조 확인 필요.

## 2. 판매자(Seller) 블로킹 이슈

### [S-01] 식품표시광고법 — AI 생성 이미지의 허위·과장 표시 위험 ⛔ AI 이미지 기능 범위 재설계

두 가지 제약이 동시에 작동한다.

**① 오픈마켓 이미지 품질 정책**
쿠팡·네이버 스마트스토어는 대표 이미지에 실물 상품 사진, 흰 배경, 상품이 이미지 영역의 80% 이상 차지할 것을 요구한다. AI가 생성한 이상화된 이미지는 이 품질 기준을 충족하기 어렵고, 심사 거절 또는 노출 제한으로 이어질 수 있다. 단, 각 플랫폼이 "AI 생성 여부" 자체를 금지 기준으로 명시하고 있지는 않으므로, 실제 반려 기준은 품질·사실성 위반 여부로 판단된다.

**② 식품표시광고법 제8조 허위·과장 표시 금지**
농수산물 온라인 판매에서 AI가 생성한 이상적인 과일 이미지를 실제와 다른 등급·외형의 상품에 사용하면, 식품표시광고법 제8조의 허위·과장 표시에 해당한다. 이미지 자체가 AI 생성이라는 사실보다, 실제 상품과 다른 품질을 표현한다는 점이 위반의 핵심이다.

**영향 범위**: SEL-05 상세페이지 초안 검토·수정 내 AI 이미지 생성 기능, content-creation-agent의 이미지 생성 기능 전체

**해결 방향**
- AI 이미지 생성 기능은 마케팅 콘텐츠 전용(SNS 배너, 썸네일, 상세페이지 하단 보조 이미지)으로 사용 범위 제한.
- 대표 이미지 슬롯(쿠팡·네이버)은 판매자가 직접 촬영한 실물 사진 업로드를 강제하는 별도 필드로 분리.
- SEL-05 화면에서 이미지 업로드 영역을 "상품 실물 이미지(직접 업로드 필수)"와 "AI 보조 이미지(마케팅용)"로 명확히 구분.

### [S-02] AI기본법 제31조 — AI 생성물 표시 의무 (v1에서 오류 수정) 

> **v1에서 오류 수정**: "CSS 오버레이 워터마크는 법적 불충족" → 실제 법령과 가이드라인 확인 결과, AI기본법은 CSS 오버레이 포함 다양한 방식을 허용하며, 핵심 구분은 "서비스 내에서만 이용"이냐 "외부로 반출"이냐이다.

**현행 법령 정확한 내용 (2026.1.22 시행, 1년 이상 계도기간 운영 중)**

AI기본법은 AI 생성물 표시 방식을 다음과 같이 구분한다.
- **서비스 환경 내에서만 이용되는 경우**: UI나 로고 표출 등을 통해 유연하게 표시 가능. SEL-05 화면에서 AI가 생성한 초안을 화면 내 뱃지로 표시하는 것은 허용된다.
- **외부로 반출되는 경우 (다운로드·공유)**: 가시적 워터마크 또는 메타데이터 삽입 중 하나를 적용해야 한다. CSS 오버레이 방식도 "가시적 워터마크"에 해당하여 허용되나, 판매자가 우클릭 저장·스크린샷으로 워터마크를 제거할 경우 플랫폼 책임이 남는지 여부는 법무 확인 필요.

**영향 범위**: SEL-05 상세페이지 초안 검토·수정, SEL-12 마케팅 초안 생성

**해결 방향**
- 서비스 UI 내에서는 "AI 생성" 텍스트 뱃지 표시로 충분.
- 외부 다운로드 기능을 제공하는 경우, 다운로드 파일에 가시적 워터마크 또는 메타데이터 삽입 처리 필요.
- 서비스 이용약관·계약서에 AI 기반 서비스임을 명시.

### [S-03] 농수산물원산지표시법 — AI 생성 텍스트의 원산지 누락·오표기 위험 ⚠️ AI 생성 범위 제한 필요

농수산물원산지표시법은 통신판매를 포함한 온라인 판매에서 원산지 표시를 의무화하고 있다. AI가 상세페이지 텍스트를 자동 생성할 때 원산지를 누락하거나 오표기하면, 표시 의무 위반 책임은 **판매자**에게 귀속된다. 플랫폼의 법적 공동 책임 여부는 법문에 명시된 것이 아니며 해석의 영역이나, AI 생성 과정에서 원산지 오류가 구조적으로 발생하면 플랫폼의 서비스 설계 책임 문제로 이어질 수 있다.

**영향 범위**: SEL-05 상세페이지 초안 검토·수정

**해결 방향**
- 원산지 관련 텍스트는 AI 자동 생성 범위에서 제외하고 판매자가 직접 입력하는 필수 필드로만 처리.
- 에디터 내 원산지 입력 필드를 AI 생성 본문과 분리된 별도 필수 항목으로 구성.
- 원산지 미입력 시 "최종 승인" 버튼 비활성화.

📎 **공식 출처**

## 3. 중개자(Intermediary) 블로킹 이슈

### [I-01] 전자상거래법 — AI 생성 CS 응답 통제 시 통신판매업자 지위 전환 위험

⛔ **영향 기능: INT-12 화면 구조 재설계**

**리스크 설명**
- 중개자 화면(INT-12)에서 플랫폼이 AI 클레임 응답 초안을 생성하고 발송 흐름을 통제하면, 전자상거래법상 **통신판매중개업자**(정보만 연결)가 아닌 **통신판매업자**(거래에 직접 개입)로 판단될 수 있다. 
- 이 경우 소비자 보호 의무가 판매자가 아닌 플랫폼에 직접 귀속된다.

**영향 범위:** INT-12 클레임 최종 판단 화면 전체

**법률 판단 기준 (공정거래위원회 해석 기준)**
- 공정거래위원회는 통신판매업자·통신판매중개자 구분이 불분명한 경우 아래 3가지를 종합 판단한다.

|판단 기준|내용|
|---|---|
|① 고지 여부|사업자가 중개자에 불과하며 책임은 의뢰자에게 있음을 소비자가 알기 쉽게 고지하였는가|
|② 결제화면 표시|대금결제가 의뢰자(판매자)와의 거래임을 소비자가 충분히 인식할 수 있도록 결제화면에 표시하였는가|
|③ 수익 구조|매출이익이 직접 자신의 수익인지, 단순 중개수수료 수익인지|
**추가 판례 기준** 
- 법원은 단순히 사이트 하단에 "중개업자로서 책임이 없다"고 표시하는 것만으로는 불충분하며, 팝업화면 고지 또는 결제 등 **중요 거래절차에서** 소비자가 그 사실을 충분히 인식할 수 있도록 조치해야 한다고 판시. 
- 즉, UI 설계뿐 아니라 **소비자가 누구와 거래하는지 인식하는 전체 플로우**가 판단 기준이다.

**해결 방향**
- AI는 **초안 생성만** 수행하고, 최종 발송 주체가 반드시 **판매자 본인**임을 법적으로 명확히 하는 구조로 설계.
- INT-12 화면에서 **"플랫폼 명의 발송" 버튼 구조 불가**. 반드시 **"판매자 명의로 발송"** 구조여야 함.
- AI 초안 표시 영역(읽기 전용)과 최종 응답 입력 영역(편집 가능)을 **분리**하고, 최종 발송은 판매자가 직접 수행하는 플로우 강제.
- **"판매자 명의 발송" 버튼 구조만으로는 충분하지 않음.** 오픈마켓(쿠팡·네이버)의 CS 시스템 자체가 판매자 계정 연동인 구조여야 하며, 플랫폼이 CS 응답 내용을 상시 검수·수정하는 구조는 "CS 개입"으로 판단될 수 있어 주의 필요.
- 법무 검토를 통해 플랫폼이 CS 응답 초안을 제공하는 행위의 법적 지위 **확인 선행 필요**.

**📎 공식 출처**

### [I-02] 금융 규제 — 플랫폼의 자금이동 승인 행위

⛔ **영향 기능: INT-11 송금 승인 기능 블로킹**

**리스크 설명**
- 1,000만원 이상 자금 이동을 플랫폼이 직접 승인·처리하는 구조가 반복되면, 금융감독원의 자금이동업 또는 전자금융업 감독 대상에 포함될 수 있다. 
- 이는 2FA 인증 구현 여부와 무관하게, 해당 기능을 제공할 **법적 자격 자체의 문제**다.

**영향 범위:** INT-11 승인 큐 중 송금 관련 기능 전체

**2025년 12월 전자금융거래법 개정으로 인한 상황 변화**: 법 개정 핵심 (2025.12.16 공포 / 시행 2026.12.17)
- 2025년 12월 개정 전자금융거래법은 전자지급결제대행(PG)의 정의에서 **"전자상거래법에 따른 통신판매중개 등 다른 업무를 영위하는 과정에서 부수적으로 대가를 수수하고 정산을 대행하는 경우"를 PG업 범위에서 명시적으로 제외**하였다.
- 이에 따라 본 플랫폼이 통신판매중개업을 영위하면서 부수적으로 정산을 대행하는 구조라면, **개정 법 시행 이후(2026.12.17~)에는 PG업 등록 의무가 발생하지 않을 가능성이 높다.**

|구분|개정 전 (현재 시행 중)|개정 후 (2026.12.17 시행)|
|---|---|---|
|통신판매중개 + 부수 정산|PG업 해당 가능성 있음 (불명확)|**PG업에서 명시 제외**|
|플랫폼 직접 자금보관|위험|위험 (티메프 규제 강화 영향)|
|PG업 등록 필요 여부|법무 검토 필요|통중개 부수 정산은 불필요할 가능성|

> ⚠️ **중요:** 개정법 **시행일(2026.12.17) 이전**까지는 현행법이 적용되므로, 시행 전 개발·운영 단계에서 법무 검토는 여전히 필수.

**추가 리스크 — 티메프 사태 이후 강화된 규제
- 2025년 개정 전자금융거래법은 PG사의 정산대금 외부관리 의무화, 정산기한 내 대금지급 의무화, 정산금 유용 시 **형사처벌(10년 이하 징역 또는 1억원 이하 벌금)** 조항을 신설하였다.
- 플랫폼이 **공급사 대금을 일정 기간 보관·지연 지급하는 구조** 자체가 위메프·티몬 사태 이후 강화된 규제 환경에서 리스크 요인이 될 수 있으므로, 정산 구조 설계 시 별도 검토 필요.

**해결 방향**
- **법무법인을 통한 자금이동업 해당 여부 검토 필수** (C-03 전자금융거래법 검토와 병행).  
    단, 2025.12 개정 내용을 반영한 최신 기준으로 검토할 것.
- 검토 결과에 따라 플랫폼이 송금을 직접 처리하는 구조 대신,  
    **PG사 정산 API를 통해 PG사가 자금을 이동하고 플랫폼은 정산 데이터만 관리하는 구조로 전환** 고려.  
    (법적 의무는 아닐 수 있으나, 리스크 헷지 및 운영 안전성 측면에서 권장)
- 공급사 대금 보관 기간 최소화 및 **정산 주기 명문화** (계약서·약관 반영) 필요.
- **법무 검토 완료 전까지 INT-11 내 송금 승인 기능 개발 보류.**
