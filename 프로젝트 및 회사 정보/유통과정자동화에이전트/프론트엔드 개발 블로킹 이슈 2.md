
## 1. 공통 블로킹 이슈

### [C-01] 네이버 커머스 API — 멀티셀러 중앙 관리 구조 불가 ⛔ MVP 착수 전 필수 해결

> **v1 대비 변경**: "IP 최대 3개 제한"이라는 단순 서술에서, 플랫폼별 정책 차이와 현실적 해결 경로를 반영하여 재작성.

**플랫폼별 정책 차이**

네이버와 쿠팡은 구조가 근본적으로 다르다.

- **네이버 스마트스토어**: 애플리케이션당 허용 IP 최대 3개, 통합매니저 계정만 발급 가능, 애플리케이션 인증 만료 시 자동 휴면 처리. 판매자 100명의 토큰을 중앙 서버에서 관리하는 현재 아키텍처는 구현 자체가 불가능하다. 네이버는 2025년 하반기 "솔루션 애플리케이션" 정식 출시를 예고한 바 있으나, 현재 시점(2026.4 기준)에서 해당 경로의 운영 가능 여부와 SLA는 미확인 상태다.
- **쿠팡 Wing**: 자체개발 방식 선택 시 IP 최대 10개까지 허용되어 네이버보다 관대하다. 단, 연동업체(셀러툴) 방식으로 등록하면 1개 업체만 지정 가능하며, API 정보 수정은 월 10회 제한이 있다.

**영향 범위**: SEL-01 온보딩 전체, SEL-06 멀티채널 상품 등록, SEL-14 정산 조회

**왜 설계를 바꿔야 하는가**

네이버 스마트스토어의 경우 중앙 서버 단일 앱 방식으로 OAuth를 처리하는 구조는 IP 제한으로 인해 구현 자체가 불가능하다. 프론트엔드 온보딩 플로우 전체를 재설계해야 한다.

**해결 방향**

- 네이버: Week 0에 솔루션 파트너 등록 신청 또는 셀러 개인 앱 등록 방식으로 온보딩 UX 설계. 파트너 승인 전까지는 셀러가 직접 API 센터에서 앱을 등록하고 발급된 키를 플랫폼에 입력하는 방식만 가능.
- 쿠팡: 자체개발 방식으로 중앙 서버 IP 최대 10개 등록 구조 적용 가능 (단, API Key 6개월 유효기간 및 월 10회 수정 제한 감안한 갱신 운영 설계 필요 → [C-NEW-02] 참조).
- 두 플랫폼 모두 중앙 일괄 자동 처리가 불가능한 만큼, 셀러 채널 연동 상태를 실시간으로 표시하고 재연동 유도하는 UX가 필수.

### [C-02] 쿠팡·네이버 플랫폼 약관 — 다중 셀러 자동화 도구 허용 범위 ⛔ 서비스 존립 리스크

쿠팡과 네이버는 약관상 단일 앱·도구로 다수 셀러 계정을 자동 관제하는 행위와 가격 자동 조정 도구 운용을 제한한다. OpenAnt의 핵심 기능인 멀티채널 자동 상품 등록, KAMIS 연동 자동 가격 조정이 이에 해당할 수 있다.

**영향 범위**: SEL-03 판매 요청 자동 생성, SEL-04 가격 자동 조정, SEL-06 멀티채널 상품 등록

**왜 설계를 바꿔야 하는가**

적발 시 서비스에 연결된 모든 셀러 계정이 동시에 정지되는 파국적 리스크가 있다. 특히 가격 자동 조정 기능을 UI에서 전면 노출하는 방식은 약관 위반 여지가 크다.

**해결 방향**

- 프론트엔드 개발 착수 전 각 플랫폼 API 파트너 담당자와 허용 범위 서면 확인 필수.
- 확인 전까지 자동 가격 조정 기능은 "판매자가 직접 수동 확인 후 적용" 방식으로 UX 설계 (자동 반영 UI 구조 개발 보류).
- 상품 등록 역시 "AI가 초안 생성 + 판매자 최종 승인 후 등록" 구조를 명확히 하여 자동 등록 도구 해석을 회피.

### [C-03] 전자금융거래법 — 플랫폼 직접 정산 분배 행위 금지 ⛔ 정산 기능 전체 블로킹

> **v1 대비 변경**: "에스크로 자금 보관·분배 행위" 표현을 실제 법적 쟁점에 맞게 수정. 티메프 사태 이후 업계 확립된 해석 반영.

중앙 계좌에 판매대금이 모이고 플랫폼이 직접 분배 로직을 통제하는 구조는, 전자금융거래법상 전자지급결제대행업(PG) 등록 없이 수행할 수 없는 "가맹점 대금지급 행위"에 해당한다. 2025년 티메프(위메프·티몬) 사태 이후 금융당국이 이 쟁점을 전면 부각시켰으며, 업계 내 확립된 해석은 다음과 같다: "플랫폼이 직접 정산을 수행할 경우 허가받지 않은 지급업무를 하는 셈이 되므로, 반드시 지급대행 라이선스를 가진 PG사를 통해 정산을 처리해야 한다."

**영향 범위**: INT-14 정산 정책 설정, SEL-14 정산 현황 조회, SUP-12 정산 내역 확인 — 정산 관련 화면 전체

**왜 설계를 바꿔야 하는가**

단순히 PG사를 경유한다고 해결되지 않는다. 플랫폼이 분배 로직 자체를 통제하는 구조(플랫폼 계좌 → 분배 → 공급사 계좌)가 규제 대상이다. 법무 검토 없이 정산 관련 화면 개발을 시작할 수 없다.

**해결 방향**

- Week 0에 법무법인을 통한 전자금융거래법 해당 여부 검토 필수.
- 검토 결과에 따라 토스페이먼츠·KG이니시스 등 PG사의 "지급대행 서비스"를 활용하여 PG사가 자금을 이동하고 플랫폼은 정산 데이터만 관리하는 구조로 전환 필요. 이 경우 플랫폼은 전자지급결제대행업 등록 없이도 합법적으로 정산 처리 가능.
- 정산 관련 화면은 법무 검토 완료 전까지 개발 보류.

### [C-04] 개인정보보호법 — 쿠팡 API 개인정보 제한 강화 및 위수탁 계약 필수 ⛔ 공급자 주문 화면 블로킹

> **v1 대비 변경**: 쿠팡 개인정보 대규모 유출 사건(2025.11, 3370만 계정)과 이후 쿠팡 API 개인정보 축소 조치를 반영하여 실질적 위험 수준 대폭 상향.

**현황 (2026년 4월 기준)**

쿠팡은 이미 API 응답에서 개인정보를 단계적으로 제거하고 있다. 2024년 8월부터 발주서 조회 API 응답에서 구매자 이메일을 빈값으로 변경했고, 2024년 9월부터는 교환요청 목록에서 실번호를 안심번호로 대체했으며, 2026년 3월에는 고객 문의 API 응답에서 `buyerEmail` 필드 자체를 삭제했다. 2025년 11월 쿠팡 3370만 계정 유출 사건 이후 개인정보보호위원회·과기정통부의 감독이 극도로 강화된 상황이다.

공급자에게 주문을 배정할 때 고객의 배송지·이름·연락처가 플랫폼 중앙 PostgreSQL을 경유하는 구조는, 쿠팡·네이버와 개인정보 처리 위수탁 계약 없이는 무단 제3자 제공에 해당한다.

**영향 범위**: SUP-06 주문 수신, SUP-08 출고 처리 — 배송 정보가 표시되는 모든 공급자 화면

**왜 설계를 바꿔야 하는가**

단순 안내 문구로 해결되는 문제가 아니다. 쿠팡 유출 사건 이후 감독당국이 배송지 정보 유통 경로를 집중 조사하고 있는 시점에서, 계약 없이 배송지 정보를 중앙 서버에 적재하는 행위는 법적 책임이 매우 크다. 또한 쿠팡이 API에서 개인정보를 지속 제거하는 추세이므로, 중앙 서버 경유 구조로 설계하더라도 향후 API 변경으로 데이터를 받지 못하는 상황이 올 수 있다.

**해결 방향**

- 배송지 정보를 중앙 서버에 저장하지 않고, 공급자 엣지 박스가 오픈마켓 API에서 직접 조회하는 방식으로 아키텍처 재설계.
- 쿠팡·네이버와 개인정보 처리 위수탁 계약 체결 또는 각 플랫폼 파트너 API 이용 약관 내 위수탁 허용 조항 확인.
- 계약 체결 전까지 SUP-06·08 화면에서 배송지 상세 정보(이름, 주소, 연락처) 노출 불가. 주문번호·상품명만 표시하는 제한 버전으로 개발.

### [C-05] 공정거래법 — 비딩 구조의 담합 촉진 가능성 ⛔ 공급자 성과·비딩 화면 블로킹

공급자 성과 대시보드(SUP-11), 배분 결과 확인 화면(INT-06)에서 낙찰 이력·타 공급자 대비 가격 순위 등이 노출되면, 공급자들이 경쟁사 가격을 역산할 수 있다. 이는 공정거래법상 가격 담합 촉진 도구로 해석될 수 있다.

**영향 범위**: SUP-11 성과 조회, INT-06 배분 결과 확인, INT-04·05 비딩 회수·시세 비교

**왜 설계를 바꿔야 하는가**

공급자에게 보여줄 수 있는 정보의 범위가 법무 검토 없이 결정될 수 없다.

**해결 방향**

- 법무 검토를 통해 공급자에게 허용 가능한 정보 범위 명확히 정의 (원칙: 자신의 응찰 이력만 허용, 타사 가격·순위 완전 차단).
- 허용 범위 확정 전까지 SUP-11, INT-06 화면의 비딩 비교 관련 UI 개발 보류.

## 2. 판매자(Seller) 블로킹 이슈

### [S-01] 식품표시광고법 + 쿠팡·네이버 이미지 정책 — AI 생성 이미지 상품 대표 이미지 사용 불가 ⛔ AI 이미지 기능 범위 재설계

두 가지 제약이 동시에 작동한다.

**① 오픈마켓 이미지 정책**

쿠팡 이미지 가이드라인은 실제 상품 사진을 요구하며, AI 합성·생성 이미지는 상품 심사 거절 사유다. 네이버 스마트스토어도 동일하게 실물 상품 이미지를 요구한다. AI가 생성한 이미지를 상품 대표 이미지로 업로드하면 상품 등록 자체가 반려된다.

**② 식품표시광고법**

농수산물 온라인 판매 시 상품 이미지는 실제 판매 상품과 동일한 외형·등급·원산지를 표현해야 한다. AI가 생성한 이상적인 과일 이미지를 실제와 다른 등급의 상품에 사용하면 허위·과장 표시 위반이다.

**영향 범위**: SEL-05 상세페이지 초안 검토·수정 내 AI 이미지 생성 기능, content-creation-agent의 이미지 생성 기능 전체

**해결 방향**

- AI 이미지 생성 기능은 마케팅 콘텐츠 전용(SNS 배너, 썸네일, 상세페이지 하단 보조 이미지)으로 사용 범위 제한.
- 상품 대표 이미지(쿠팡 메인 이미지 슬롯, 네이버 대표 이미지 슬롯)는 판매자가 직접 촬영한 사진 업로드를 강제하는 별도 필드로 분리.
- SEL-05 화면에서 이미지 업로드 영역을 "상품 실물 이미지(직접 업로드 필수)"와 "AI 보조 이미지(마케팅용)"로 명확히 구분.

### [S-02] AI기본법 제31조 — AI 생성물 표시 의무 (v1에서 오류 수정) ⚠️ 법적 대응 필요

> **v1에서 오류 수정**: "CSS 오버레이 워터마크는 법적 불충족" → 실제 법령과 가이드라인 확인 결과, AI기본법은 CSS 오버레이 포함 다양한 방식을 허용하며, 핵심 구분은 "서비스 내에서만 이용"이냐 "외부로 반출"이냐이다.

**현행 법령 정확한 내용 (2026.1.22 시행, 1년 이상 계도기간 운영 중)**

AI기본법은 AI 생성물 표시 방식을 다음과 같이 구분한다.

- **서비스 환경 내에서만 이용되는 경우**: UI나 로고 표출 등을 통해 유연하게 표시 가능. 예를 들어 SEL-05 화면에서 AI가 생성한 상세페이지 초안을 화면 내 뱃지로 표시하는 것은 허용된다.
- **외부로 반출되는 경우 (다운로드·공유)**: 사람이 인식할 수 있는 방법(가시적 워터마크 등) 또는 기계가 판독할 수 있는 방법(메타데이터 삽입 등) 중 하나를 적용해야 한다. CSS 오버레이 방식도 "가시적 워터마크"에 해당하여 허용되나, 판매자가 우클릭 저장·스크린샷으로 워터마크를 제거할 경우 플랫폼 책임이 남는지 여부는 법무 확인 필요.

**영향 범위**: SEL-05 상세페이지 초안 검토·수정, SEL-12 마케팅 초안 생성

**실질적 위험 (v1보다 낮지만 여전히 대응 필요)**

계도기간 중이라 과태료(최대 3천만원) 부과 가능성은 낮으나, 서비스 출시 후 법 적용 시 즉시 대응할 수 있도록 설계 단계에서 고려해야 한다.

**해결 방향**

- 서비스 UI 내에서는 "AI 생성" 텍스트 뱃지 표시(가시적 표시)로 충분.
- AI 생성 이미지·텍스트를 외부 다운로드할 수 있는 기능을 제공하는 경우, 다운로드 파일에 가시적 워터마크 또는 메타데이터 삽입 처리 필요.
- v1의 "백엔드 스테가노그래피 방식 필수" 주장은 과도하게 엄격한 해석임. 현행 가이드라인상 불필요하다. 다만 향후 법령 강화에 대비하여 백엔드 메타데이터 삽입 방식을 중장기 로드맵에 포함.
- 서비스 이용약관·계약서에 AI 기반 서비스임을 명시 (사전 고지 의무 이행).

---

### [S-03] 농수산물원산지표시법 — AI 생성 상세페이지 텍스트의 원산지 오표기 시 플랫폼 공동 책임 ⛔ AI 생성 범위 제한 필요

AI가 상세페이지 텍스트를 자동 생성할 때 원산지를 누락하거나 오표기할 경우, 플랫폼이 생성 행위에 관여했으므로 허위·과장 표시에 대한 공동 책임이 발생한다.

**영향 범위**: SEL-05 상세페이지 초안 검토·수정

**해결 방향**

- 원산지 관련 텍스트는 AI 자동 생성 범위에서 완전히 제외하고 사람이 직접 입력하는 필드로만 처리 (백엔드 프롬프트 설계 요구사항으로 전달).
- 에디터 내 원산지 입력 필드를 AI 생성 본문과 완전히 분리된 별도 필수 항목으로 구성.
- 원산지 미입력 시 "최종 승인" 버튼 비활성화.

## 3. 중개자(Intermediary) 블로킹 이슈

### [I-01] 전자상거래법 — AI 생성 CS 응답 통제 시 통신판매업자 지위 전환 위험 ⛔ INT-12 화면 구조 재설계

중개자 화면(INT-12)에서 플랫폼이 AI 클레임 응답 초안을 생성하고 발송 흐름을 통제하면, 전자상거래법상 통신판매중개업자(정보만 연결)가 아닌 통신판매업자(거래에 직접 개입)로 판단될 수 있다. 이 경우 소비자 보호 의무가 판매자가 아닌 플랫폼에 직접 귀속된다.

**영향 범위**: INT-12 클레임 최종 판단 화면 전체

**해결 방향**

- AI는 초안 생성만 수행하고, 최종 발송 주체가 반드시 판매자 본인임을 법적으로 명확히 하는 구조로 설계.
- INT-12 화면에서 "플랫폼 명의 발송" 버튼 구조 불가. 반드시 "판매자 명의로 발송" 구조여야 함.
- AI 초안 표시 영역(읽기 전용)과 최종 응답 입력 영역(편집 가능)을 분리하고, 최종 발송은 판매자가 직접 수행하는 플로우 강제.
- 법무 검토를 통해 플랫폼이 CS 응답 초안을 제공하는 행위의 법적 지위 확인 선행 필요.

### [I-02] 금융 규제 — 플랫폼의 자금이동 승인 행위 ⛔ INT-11 송금 승인 기능 블로킹

1,000만원 이상 자금 이동을 플랫폼이 직접 승인·처리하는 구조가 반복되면, 금융감독원의 자금이동업 또는 전자금융업 감독 대상에 포함될 수 있다. 이는 2FA 인증 구현 여부와 무관하게, 해당 기능을 제공할 법적 자격 자체의 문제다.

**영향 범위**: INT-11 승인 큐 중 송금 관련 기능 전체

**해결 방향**

- 법무법인을 통한 자금이동업 해당 여부 검토 필수 (C-03 전자금융거래법 검토와 병행).
- 검토 결과에 따라 플랫폼이 송금을 직접 처리하는 구조 대신, PG사 정산 API를 통해 PG사가 자금을 이동하고 플랫폼은 정산 데이터만 관리하는 구조로 전환 필요.
- 법무 검토 완료 전까지 INT-11 내 송금 승인 기능 개발 보류.

### [C-NEW-01] 쿠팡 API Rate Limit — 초당 10회 초과 즉시 IP·VendorID 차단 ⛔ 발주 폴링 아키텍처 재설계

쿠팡 Open API는 초당 10회 이상 호출 시 즉시 차단을 적용한다. 5분 폴링 주기로 다수 셀러 계정을 순차 조회하면 기준을 쉽게 초과할 수 있으며, 차단 시 해당 IP와 VendorID 전체가 차단되어 모든 셀러 주문 수신이 중단된다.

**영향 범위**: SEL-08 주문 현황 조회, INT-09 주문 배분 통제 (5분 폴링 구조 전체)

**해결 방향**

- 셀러 수 × 채널 수 × API 호출 수를 계산하여 초당 10회 이하로 설계. 셀러가 30명이면 5분(300초)에 30회 호출로 초당 0.1회 — 안전. 셀러 100명 이상부터는 Queue + Rate Limiter 설계 필수.
- API 호출 오류 로그 실시간 모니터링 및 자동 알림 구축 필수.
- 프론트엔드에서는 "마지막 폴링 시각 + 오류 여부"를 INT-10 대시보드에 명시적으로 표시.

---

### [C-NEW-02] 쿠팡·네이버 API 키 유효기간 만료 — 서비스 자동 중단 리스크 ⛔ 운영 자동화 필수

- **쿠팡**: Open API Key 유효기간 최대 6개월. 만료 전 갱신하지 않으면 해당 셀러 전체 API 기능 중단.
- **네이버**: 앱 인증 만료 시 자동 휴면 처리. 인증 기한이 애플리케이션별로 표시되며, 만료 전 반드시 수동 인증 완료 필요.

**영향 범위**: SEL-01 계정 연동, SEL-06·07 상품 등록·조회 — 키 만료 시 전체 기능 중단

**해결 방향**

- 쿠팡 API Key 만료 30일·7일·1일 전 운영자 및 해당 셀러에게 자동 알림 발송 기능 구현.
- 네이버 앱 인증 만료 알림 및 재인증 유도 UX 구현.
- INT-10 운영 모니터링 대시보드에 채널별 API 키 만료일 표시 필수 (현재 기능 명세 누락).
- SEL-01 온보딩 화면에 "키 유효기간 안내 및 갱신 방법" 안내 문구 포함.

### [C-NEW-03] 쿠팡 개인정보 유출 사건 후속 규제 강화 — API 배송 정보 접근 추가 제한 예고 ⚠️ 지속 모니터링

2025년 11월 쿠팡 3370만 계정 개인정보 유출 사건 이후, 쿠팡은 2026년 3월에 고객 문의 API에서 buyerEmail 필드를 삭제했다. 감독당국이 배송지 정보 유통 경로를 집중 조사 중이며, 발주서 조회 API에서 배송지 상세 정보(이름, 주소, 전화번호)가 추가 제한될 가능성이 있다.

**영향 범위**: SUP-06 주문 수신 (배송지 기반 출고 처리 전체)

**해결 방향**

- 쿠팡 Open API 공지사항(developers.coupangcorp.com)을 월 1회 정기 모니터링하여 API 변경 사전 파악.
- 배송지 정보를 중앙 DB에 캐싱하는 구조 지양, 공급자가 API에서 직접 조회하는 방식 채택 (C-04 해결 방향과 동일).
- 배송지 정보가 API에서 제거될 경우의 대안 플로우(안심번호 기반 배송 등) 사전 설계.

### [C-NEW-04] 에이전트 서비스 법적 지위 — "단순 연결"이냐 "운영 개입"이냐 ⛔ 전체 아키텍처 관통 이슈

두 번째 문서(질의사항 정리)에서 명확히 제기된 쟁점이다. "단순 판매자·중개자·공급자 유통 에이전트인데 법적 문제가 있나? 우리 플랫폼 구독료만 받는데?"라는 질문에 대한 답은 "있다"이다.

법은 이름이 아니라 실제 역할(행위)을 기준으로 판단한다. 아래 행위 중 하나라도 플랫폼이 수행하면 "단순 연결"로 볼 수 없다.

- 상품 정보 생성 (상세페이지·카피 작성) → 정보 생산자
- 가격·마진 계산 관여 → 의사결정 영향
- CS 응답 초안 생성 또는 전달 → 고객 응대 개입
- 환불·교환 프로세스 개입 → 거래 흐름 제어
- 공급자 선정·비딩 개입 → 거래 구조 통제
- 주문·배송 흐름 제어 → 이행 주체 역할

**영향 범위**: 전체 서비스 아키텍처

**핵심 리스크**

- 소비자 책임이 플랫폼으로 전이될 수 있음
- 허위·과장 정보 책임
- CS 응답 책임
- 식품·품질 사고 책임 일부 전이 가능

**해결 방향**

- 플랫폼이 수행하는 각 행위(상세페이지 생성, 가격 추천, CS 초안, 배분 계산 등)에 대해 법무법인과 함께 "정보 제공"과 "운영 개입"의 경계를 명확히 획정.
- Week 0 법무 검토의 최우선 항목으로 설정.
- 각 기능 화면에서 "AI가 생성한 초안이며 최종 실행은 사용자 책임" 고지를 단순 면책 문구가 아닌 구조적 UX로 구현 (승인 게이트, 별도 입력 필드 분리 등).
