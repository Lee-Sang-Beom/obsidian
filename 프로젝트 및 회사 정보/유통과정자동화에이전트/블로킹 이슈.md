## 1. 공통 블로킹 이슈

### [C-01] 네이버 커머스 API — 멀티셀러 IP 제한 ⛔ MVP 착수 전 필수 해결

> 참고: https://notice.naver.com/notices/cac/16929

네이버 커머스 API는 **애플리케이션당 허용 IP를 최대 3개**로 제한한다. 판매자 100명의 토큰을 중앙 서버 하나에서 관리하는 현재 아키텍처는, 셀러 수가 늘어나는 순간 네이버 API 전체가 차단된다.

**영향 범위**: SEL-01 온보딩 전체, 멀티채널 상품 등록 전체, 정산 조회

**왜 설계를 바꿔야 하는가**

현재 기획은 중앙 서버 단일 앱 방식으로 OAuth를 처리하는 구조를 전제한다. 이 구조는 IP 제한으로 인해 구현 자체가 불가능하다. 프론트엔드 온보딩 플로우 전체를 **"셀러 개인 앱 등록 → 토큰 위탁"** 방식으로 재설계해야 하며, 중앙 서버 단일 앱 전제의 OAuth 리다이렉트 플로우는 설계할 수 없다.

**해결 방향**

- Week 0에 네이버 커머스 솔루션 파트너 등록 신청 (멀티셀러 구조 공식 지원)
- 파트너 승인 전까지는 셀러가 직접 네이버 API 센터에서 앱을 등록하고 토큰을 발급받아 플랫폼에 입력하는 방식으로 온보딩 UX 설계
- 파트너 승인 완료 후 중앙 OAuth 플로우로 전환하는 2단계 설계 필요

### [C-02] 쿠팡·네이버 플랫폼 약관 — 다중 셀러 자동화 도구 금지 ⛔ 서비스 존립 리스크

쿠팡과 네이버는 약관상 단일 앱·도구로 다수 셀러 계정을 자동 관제하는 행위와 가격 자동 조정 도구 운용을 제한한다. OpenAnt의 핵심 기능인 멀티채널 자동 상품 등록, KAMIS 연동 자동 가격 조정이 이에 해당할 수 있다.

**영향 범위**: SEL-03 판매 요청 자동 생성, SEL-04 가격 자동 조정, SEL-06 멀티채널 상품 등록

**왜 설계를 바꿔야 하는가**

적발 시 서비스에 연결된 **모든 셀러 계정이 동시에 정지**되는 파국적 리스크가 있다. 특히 가격 자동 조정 기능을 UI에서 전면 노출하는 방식은 약관 위반 여지가 크다.

**해결 방향**

- 프론트엔드 개발 착수 전 각 플랫폼 API 파트너 담당자와 허용 범위 서면 확인 필수
- 확인 전까지 자동 가격 조정 기능은 "판매자가 직접 수동 확인 후 적용" 방식으로 UX 설계 (자동 반영 UI 구조 개발 보류)
- 상품 등록 역시 "AI가 초안 생성 + 판매자 최종 승인 후 등록" 구조를 명확히 하여 자동 등록 도구 해석을 회피

### [C-03] 전자금융거래법 — 에스크로 자금 보관·분배 행위 ⛔ 정산 기능 전체 블로킹

중앙 에스크로 계정에 판매대금이 적립되고 플랫폼이 분배 로직을 통제하는 구조는, **전자금융거래법상 전자금융업 등록이 필요한 행위**에 해당할 수 있다. 미등록 시 불법 전자금융업으로 처벌받는다.

**영향 범위**: INT-14 정산 정책 설정, SEL-14 정산 현황 조회, SUP-12 정산 내역 확인 — 정산 관련 화면 전체

**왜 설계를 바꿔야 하는가**

PG사 에스크로 API를 경유하더라도 플랫폼이 분배를 통제하는 구조 자체가 규제 대상이 될 수 있다. 법무 검토 없이 정산 관련 화면 개발을 시작할 수 없다.

**해결 방향**

- Week 0에 법무법인을 통한 전자금융거래법 해당 여부 검토 필수
- 검토 결과에 따라 PG사(토스페이먼츠·KG이니시스)의 에스크로 서비스를 활용하고 플랫폼은 분배 로직 통제에서 배제하는 구조로 전환 필요
- 정산 관련 화면은 법무 검토 완료 전까지 개발 보류

### [C-04] 개인정보보호법 — 고객 주문 데이터(배송지·이름·연락처) 중앙 서버 경유 ⛔ 공급자 주문 화면 블로킹

공급자에게 주문을 배정할 때 고객의 배송지·이름·연락처가 중앙 PostgreSQL을 경유한다. 쿠팡·네이버와 플랫폼 간 개인정보 처리 **위수탁 계약이 없으면 무단 제3자 제공**에 해당한다.

**영향 범위**: SUP-06 주문 수신, SUP-08 출고 처리 — 배송 정보가 표시되는 모든 공급자 화면

**왜 설계를 바꿔야 하는가**

단순 안내 문구로 해결되는 문제가 아니다. 데이터 흐름 자체를 바꿔야 한다. 현재 아키텍처대로 개발하면 개인정보보호법 위반 상태로 서비스가 운영된다.

**해결 방향**

- 배송지 정보를 중앙 서버에 저장하지 않고, 공급자 에이전트(Mac Mini)가 오픈마켓 API에서 직접 조회하는 방식으로 아키텍처 재설계
- 쿠팡·네이버와 개인정보 처리 위수탁 계약 체결 또는 각 플랫폼의 파트너 API 이용 약관 내 위수탁 허용 조항 확인
- 계약 체결 전까지 SUP-06·08 화면에서 배송지 상세 정보 노출 불가 (주문번호·상품명만 표시하는 제한 버전으로 개발)

### [C-05] 공정거래법 — 비딩 구조의 담합 촉진 가능성 ⛔ 공급자 성과·비딩 화면 블로킹

공급자 성과 대시보드(SUP-11), 배분 결과 확인 화면(INT-06)에서 낙찰 이력·타 공급자 대비 가격 순위 등이 노출되면, 공급자들이 경쟁사 가격을 역산할 수 있다. 이는 **공정거래법상 가격 담합 촉진 도구**로 해석될 수 있다.

**영향 범위**: SUP-11 성과 조회, INT-06 배분 결과 확인, INT-04·05 비딩 회수·시세 비교

**왜 설계를 바꿔야 하는가**

공급자에게 보여줄 수 있는 정보의 범위가 법무 검토 없이 결정될 수 없다. 현재 기획대로 개발하면 담합 촉진 도구를 플랫폼이 제공하는 구조가 된다.

**해결 방향**

- 법무 검토를 통해 공급자에게 허용 가능한 정보 범위 명확히 정의 (원칙: 자신의 응찰 이력만 허용, 타사 가격·순위 완전 차단)
- 허용 범위 확정 전까지 SUP-11, INT-06 화면의 비딩 비교 관련 UI 개발 보류

## 2. 판매자(Seller) 블로킹 이슈

### [S-01] 식품표시광고법 + 쿠팡·네이버 이미지 정책 — AI 생성 이미지 상품 대표 이미지 사용 불가 ⛔ AI 이미지 기능 범위 재설계

두 가지 제약이 동시에 작동한다.

**① 오픈마켓 이미지 정책**

쿠팡 이미지 가이드라인은 실제 상품 사진을 요구하며, AI 합성·생성 이미지는 상품 심사 거절 사유다. 네이버 스마트스토어도 동일하게 실물 상품 이미지를 요구한다. AI가 생성한 이미지를 상품 대표 이미지로 업로드하면 상품 등록 자체가 반려된다.

**② 식품표시광고법**

농수산물 온라인 판매 시 상품 이미지는 실제 판매 상품과 동일한 외형·등급·원산지를 표현해야 한다. AI가 생성한 이상적인 과일 이미지를 실제와 다른 등급의 상품에 사용하면 허위·과장 표시 위반이다.

**영향 범위**: SEL-05 상세페이지 초안 검토·수정 내 AI 이미지 생성 기능, content-creation-agent의 이미지 생성 기능 전체

**왜 설계를 바꿔야 하는가**

Stable Diffusion 기반 AI 이미지 생성 기능을 상품 대표 이미지 용도로 제공하는 것 자체가 불가능하다. 이 기능이 MVP 핵심 기능으로 기획되어 있으므로 범위 재정의가 필요하다.

**해결 방향**

- AI 이미지 생성 기능은 **마케팅 콘텐츠 전용**(SNS 배너, 썸네일, 상세페이지 하단 보조 이미지)으로 사용 범위 제한
- 상품 대표 이미지(쿠팡 메인 이미지 슬롯, 네이버 대표 이미지 슬롯)는 **판매자가 직접 촬영한 사진 업로드를 강제**하는 별도 필드로 분리
- SEL-05 화면에서 이미지 업로드 영역을 "상품 실물 이미지(직접 업로드 필수)"와 "AI 보조 이미지(마케팅용)"로 명확히 구분

### [S-02] AI기본법 제31조 — CSS 오버레이 워터마크의 법적 불충족 ⛔ 백엔드 이미지 파이프라인 재설계 필요

현재 기획은 AI 생성 이미지에 CSS 오버레이로 반투명 워터마크를 렌더링하는 방식을 제안한다. 그러나 판매자가 이미지를 우클릭 저장 또는 스크린샷으로 다운로드하면 워터마크가 제거된다.

**왜 설계를 바꿔야 하는가**

AI기본법이 요구하는 **비가시적 워터마크(메타데이터 삽입 또는 스테가노그래피 방식)**는 이미지 생성 단계에서 픽셀 수준으로 삽입되어야 한다. 이는 프론트엔드로 해결되는 문제가 아니다.

**해결 방향**

- 프론트엔드는 "AI 생성" 텍스트 배지 표시(가시적 표시)만 담당
- 비가시적 워터마크는 백엔드 이미지 생성 파이프라인(content-creation-agent) 단계에서 처리하도록 백엔드 요구사항으로 전달
- CSS 오버레이 워터마크 방식은 법적 요건 미충족으로 제거

### [S-03] 농수산물원산지표시법 — AI 생성 상세페이지 텍스트의 원산지 오표기 시 플랫폼 공동 책임 ⛔ AI 생성 범위 제한 필요

AI가 상세페이지 텍스트를 자동 생성할 때 원산지를 누락하거나 오표기(예: "국내산" 고정 삽입)할 경우, 플랫폼이 생성 행위에 관여했으므로 **허위·과장 표시에 대한 공동 책임**이 발생한다.

**영향 범위**: SEL-05 상세페이지 초안 검토·수정

**왜 설계를 바꿔야 하는가**

단순히 원산지 필드를 필수 입력으로 두는 것만으로 부족하다. AI 생성 본문에 원산지가 이미 잘못 포함되어 있을 경우, 판매자가 이를 인지하지 못하고 그대로 승인할 수 있다.

**해결 방향**

- 원산지 관련 텍스트는 AI 자동 생성 범위에서 완전히 제외하고 **사람이 직접 입력하는 필드로만 처리** (백엔드 프롬프트 설계 요구사항으로 전달)
- 에디터 내 원산지 입력 필드를 AI 생성 본문과 완전히 분리된 별도 필수 항목으로 구성
- 원산지 미입력 시 "최종 승인" 버튼 비활성화

## 3. 중개자(Intermediary) 블로킹 이슈

### [I-01] 전자상거래법 — AI 생성 CS 응답 통제 시 통신판매업자 지위 전환 위험 ⛔ INT-12 화면 구조 재설계

중개자 화면(INT-12)에서 플랫폼이 AI 클레임 응답 초안을 생성하고 발송 흐름을 통제하면, 전자상거래법상 "통신판매중개업자"(정보만 연결)가 아닌 **"통신판매업자"(거래에 직접 개입)**로 판단될 수 있다. 이 경우 소비자 보호 의무가 판매자가 아닌 플랫폼에 직접 귀속된다.

**영향 범위**: INT-12 클레임 최종 판단 화면 전체

**왜 설계를 바꿔야 하는가**

이는 UX 설계의 문제가 아니라 **플랫폼의 법적 지위를 결정하는 구조적 설계 문제**다. 플랫폼이 CS 응답 발송 주체로 보이는 순간 법적 책임 구조가 전환된다.

**해결 방향**

- AI는 초안 생성만 수행하고, 최종 발송 주체가 반드시 **판매자 본인**임을 법적으로 명확히 하는 구조로 설계
- INT-12 화면에서 "플랫폼 명의 발송" 버튼 구조 불가. 반드시 "판매자 명의로 발송" 구조여야 함
- AI 초안 표시 영역(읽기 전용)과 최종 응답 입력 영역(편집 가능)을 분리하고, 최종 발송은 판매자가 직접 수행하는 플로우 강제
- 법무 검토를 통해 플랫폼이 CS 응답 초안을 제공하는 행위의 법적 지위 확인 선행 필요

---

### [I-02] 금융 규제 — 플랫폼의 1,000만원 이상 자금이동 승인 행위 ⛔ INT-11 송금 승인 기능 블로킹

1,000만원 이상 자금 이동을 플랫폼이 직접 승인·처리하는 구조가 반복되면, 금융감독원의 **자금이동업 또는 전자금융업 감독 대상**에 포함될 수 있다. 이는 2FA 인증 구현 여부와 무관하게, 해당 기능을 제공할 법적 자격 자체의 문제다.

**영향 범위**: INT-11 승인 큐 중 송금 관련 기능 전체

**왜 설계를 바꿔야 하는가**

2FA는 보안 구현의 문제다. 그 이전에 플랫폼이 이 기능을 제공할 수 있는 법적 자격이 있는지를 먼저 확인해야 한다. 자격 없이 개발·운영하면 금융법 위반이다.

**해결 방향**

- 법무법인을 통한 자금이동업 해당 여부 검토 필수 (C-03 전자금융거래법 검토와 병행)
- 검토 결과에 따라 플랫폼이 송금을 직접 처리하는 구조 대신, PG사 정산 API를 통해 PG사가 자금을 이동하고 플랫폼은 정산 데이터만 관리하는 구조로 전환 필요
- 법무 검토 완료 전까지 INT-11 내 송금 승인 기능 개발 보류
