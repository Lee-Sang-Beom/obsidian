```
현재 프론트엔드 프로젝트에서 멤버십 결제 플랜 구조를 전면 개편하는 리팩토링을 진행하려고 해. 결제와 관련된 로직이 여러 파일에 걸쳐 복잡하게 얽혀 있으므로, 아래의 변경 사항과 제약 조건을 완벽하게 숙지하고 관련된 모든 파일(types, constants, hooks, UI components, utils)을 수정해 줘.1. 변경 사항 (AS-IS ➔ TO-BE)•기존(AS-IS): 무료 회원 / 베이직(월간, 6개월) / 프리미엄(월간, 6개월)•변경(TO-BE): 무료 회원 / PRO(월간)•즉, basic, premium이라는 개념과 타입은 모두 사라지고 pro 1개로 통합해 줘.•semiannually(6개월) 개념과 변수명도 모두 지우고 결제 주기는 monthly(월간)만 남겨 줘.2. 절대 지켜야 할 제약 조건 (Critical Constraints)•결제 코어 로직 보존: 서버 API 연동(initiate, verify 등)과 모바일 앱(Flutter WebView Bridge) 간의 통신 흐름(3단계 결제 프로세스)은 절대 수정하거나 삭제하지 마.•안전한 간소화: 플랜이 단일화되면서 불필요해진 로직(예: 업그레이드/다운그레이드 분기 처리, 6개월 할인 배지 UI, 복잡한 권한 체크 로직)은 과감히 제거하되, 결제 흐름 자체가 끊기지 않도록 매끄럽게 연결해 줘.3. 작업 지시 사항•types.ts, constants.ts부터 시작해서 관련된 hooks, utils, components를 모두 탐색(grep 등 활용)하여 일관성 있게 수정해 줘.•작업이 끝나면 프론트엔드 계층별(도메인, 훅, UI, 유틸)로 어떤 부분을 어떻게 간소화했는지 요약해서 알려줘.
```
